# Business Rules Baseline

## 目标

本文件是当前系统关键业务规则的正式稳定入口。  
如与 `docs/reference/` 中的原始输入材料存在冲突，应回到原始题面和实现代码核对；日常开发默认以本文件作为工作入口。

## 学术空间预约

- 资源单元是原子资源，不可拆分共享
- 用户选择连续时间段提交预约
- 冲突判断使用实际占用区间：
  - `startTime - 5min`
  - `endTime + 5min`
- 同一资源单元在有效预约状态下不得重叠
- 预约创建时会同时生成统一订单
- 订单取消或超时后，预约状态必须同步释放

## 体育设施预约

- 采用 `1` 小时离散槽位模型
- 允许：
  - 单资源单元预约
  - 组合资源预约
- 组合预约只要任一资源单元任一槽位冲突，则整单失败
- 同一资源单元同一槽位在有效状态下只能被一个订单占用
- 订单取消或超时后，槽位状态必须同步释放

## 活动报名与抢票

- 只有 `PUBLISHED` 且位于报名时间窗口内的活动允许抢票
- 抢票入口先经过 Redis 热点校验和库存预扣
- 用户对同一活动只能保留一条有效报名
- 异步 worker 落库时仍需做数据库最终库存校验
- 建单失败时必须回补 Redis 预扣库存
- 订单取消后应回写活动报名状态并释放票种占用

## 订单状态机

当前统一订单状态为：

- `PENDING_CONFIRMATION`
- `CONFIRMED`
- `CANCELLED`
- `NO_SHOW`

当前正式边界：

- 普通用户：
  - 查看自己的订单
  - 取消允许取消的订单
- 管理员：
  - 确认订单
  - 标记爽约
- 内部任务：
  - 执行超时取消

所有关键迁移要求：

- 写入 `OrderStatusLog`
- 基于 `version` 做 CAS 保护
- 关联业务实体同步更新状态

## 幽灵支付保护

- 超时取消通过 delayed job 异步执行，而不是依赖手工 HTTP 触发
- 支付或后续确认动作必须基于旧状态校验
- 已取消订单不得被迟到任务重新确认

## 规则引擎

当前正式支持的规则类型：

- `min_credit_score`
- `max_duration_minutes`
- `allowed_user_roles`

当前执行原则：

- 规则通过 `Rule + ResourceRuleBinding + UserRuleProfile` 表达
- 规则校验通过统一执行器进入预约主流程
- 主流程不继续堆叠分散的硬编码 if-else

## 验证与维护

- 业务规则变更时，应同步更新：
  - 本文件
  - 相关验证证据
  - `feature-list.json` 中对应验收项
- 若变更已影响领域模型，还应同步更新：
  - `docs/architecture/domain-model-baseline.md`
  - `apps/api/prisma/schema.prisma`
