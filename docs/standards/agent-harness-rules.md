# Agent Harness Rules

## 目标

将仓库组织成适合长周期 agent 持续工作的执行环境，降低跨会话漂移、重复探索和误判完成状态的风险。

## 核心原则

- 仓库是 agent 协作的主记忆体，长期有效的信息必须版本化保存
- `AGENTS.md` 只承担入口、约束和导航职责，不承载大段细则
- 规则、计划、进度、决策分层存放，避免混写
- 每次只推进一个当前最高优先级且边界清晰的目标
- 任何“已完成”都必须建立在验证结果之上，而不是主观判断

## 开始工作前

- 阅读 `docs/progress/agent-progress.md`
- 阅读 `docs/plans/feature-list.json`
- 阅读最近的 git 历史，确认当前基线
- 识别本次只推进的一个目标
- 如果任务较大，先输出方案并等待批准

## 执行中

- 优先把关键信息写回仓库，而不是停留在聊天记录
- 优先补导航、边界和结构，再补大段说明
- 遇到重复性错误时，补规则、模板或脚本，不只补口头提醒
- 不依赖隐性上下文，例如个人记忆、外部文档链接或临时聊天结论
- 修改范围保持最小化，避免一次引入多个不相关变化

## 完成判定

- 功能、规则或文档必须经过最小必要验证
- 未验证前，不更新完成状态
- 阶段结束后必须更新进度记录
- 阶段结束后必须执行 git 提交

## 文档落位

- 长期规则放在 `docs/standards/`
- 重要决策放在 `docs/adr/`
- 阶段计划放在 `docs/exec-plan/`
- 任务状态放在 `docs/plans/feature-list.json`
- 会话交接和阶段进展放在 `docs/progress/agent-progress.md`
- 原始输入材料保留在 `docs/reference/`

## 反模式

- 把关键背景只留在聊天记录里
- 在 `AGENTS.md` 中堆积大量细节
- 一次同时推进多个未完成的高优先级事项
- 没有验证就宣称完成
- 发现流程问题后只靠人工记忆维持
