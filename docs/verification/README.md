# Verification Evidence

本目录用于存放功能验证、回归验证、UI 截图、压测结果和其他可追溯证据。

## 目标

- 将“验证通过”从口头结论变成可追溯证据
- 为跨会话回归提供依据
- 为将任务状态从未通过改成通过提供凭证

## 适用场景

- 前端页面或交互流程验证
- E2E 测试结果存档
- API 自测结果存档
- 压测与性能报告
- 无障碍检查报告
- Docker、Nginx、部署验证记录

## 建议结构

- 按日期建立子目录，例如 `2026-04-16/`
- 同一天内按主题继续分目录，例如 `2026-04-16/foundation-stack/`

示例：

```text
docs/verification/
└── 2026-04-16/
    ├── foundation-stack/
    │   ├── docker-compose-build.txt
    │   ├── pnpm-build.txt
    │   └── web-home-screenshot.png
    └── reservation-flow/
        ├── booking-e2e.txt
        └── booking-page.png
```

## 证据类型

- `*.png` / `*.jpg`：页面截图
- `*.txt` / `*.log`：命令输出摘要、测试输出、部署结果
- `*.md`：人工验证记录与结论
- 外部录屏或报告链接：建议附一份同名 `*.md` 说明链接与上下文

## 使用规则

- 将功能项从未通过改为通过前，应至少留下一份对应证据
- 回归失败时，应在证据目录追加失败记录，而不是只在聊天中说明
- 证据文件名应包含日期、主题和验证动作，避免使用无意义名称
- 进度记录中应引用对应证据目录或文件
