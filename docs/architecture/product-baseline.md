# Product Baseline

## 目标

`CampusBook` 当前的正式产品目标是：在单机 `2C2G` 部署约束下，提供一个可真实交互、可演示、可持续维护的校园预约与活动平台。

当前稳定支持 3 条核心业务线：

- 学术空间预约
- 体育设施预约
- 校园活动报名与抢票

## 用户与身份

系统当前有两类正式身份：

- `STUDENT`：普通用户，可登录、查看资源与活动、提交预约/报名、查看和取消自己的订单
- `ADMIN`：管理员，可进入 `/admin`，维护资源、活动、票种与规则，并执行受管控的订单动作

## 产品范围

### 用户端

- 登录、刷新登录态、退出
- 浏览学术空间与体育设施资源
- 创建学术空间预约
- 创建体育设施单资源和组合预约
- 浏览活动、抢票、查看报名状态
- 查看自己的订单并执行允许的取消动作

### 管理端

- 创建和更新资源
- 创建资源单元与组合资源
- 创建和更新活动
- 为活动新增票种
- 查看和维护规则
- 执行管理员专属订单动作

### 平台能力

- 统一订单状态机
- Redis + BullMQ 热点削峰与异步建单
- 基于资源绑定的规则引擎
- HTTPS 部署与 live smoke 校验

## 核心用户路径

### 学术空间预约

1. 用户登录
2. 查看学术空间列表和详情
3. 提交连续时间段预约
4. 系统执行规则校验、缓冲区冲突校验和下单
5. 用户在订单页查看结果

### 体育设施预约

1. 用户登录
2. 查看体育设施、资源单元和组合资源
3. 选择离散槽位提交预约
4. 系统执行规则校验、槽位冲突校验和下单
5. 用户在订单页查看结果

### 活动报名与抢票

1. 用户登录
2. 浏览活动列表与详情
3. 选择票种发起抢票
4. 请求先进入 Redis 预扣，再异步建单
5. 用户查询报名状态或在订单页查看结果

## 非功能基线

- 部署基线：单机 Docker Compose
- 域名结构：
  - `campusbook.top`
  - `www.campusbook.top`
  - `api.campusbook.top`
- 正式环境：HTTPS 已启用
- 数据与状态要求：
  - 关键状态变更可审计
  - 关键冲突由数据库兜底
  - 热点活动必须避免超卖
- 交付要求：
  - 仓库内保留验证证据
  - CI 运行 `prisma:generate / lint / typecheck / test / build`
